# Component Guidelines

本文件定义 Demo Page 的推荐拆分方式，目标是让多个页面共享结构，而不是每页从零开始拼。

## 1. 页面级组件清单

每个 Demo Page 优先从以下组件组合：

- `DemoPageShell`
  - 负责页面容器、背景层、内容层、状态层。
- `SceneHero`
  - 负责封面、视频、场景标题、节目标识。
- `StorySummary`
  - 负责剧情摘要、角色关系、当前问题。
- `ChoiceCard` / `ActionCard`
  - 负责按钮、选项、结果反馈。
- `StatusLayer`
  - 负责 loading / fallback / error / empty 提示。
- `PageFooterAction`
  - 负责主 CTA、辅助 CTA、跳转说明。

## 2. 组件拆分原则

1. 视觉区块和业务状态分开。
2. 数据消费组件只接结构化字段，不直接拼接业务推理。
3. 通用组件不写死具体节目名称和剧情文案。
4. 页面编排组件可以引用 Task 级配置，但不能把配置散落到子组件内部。

## 3. 命名规则

- React 组件命名使用 PascalCase。
- 页面配置字段使用小写下划线或 camelCase，保持项目内一致即可，但同一目录不能混用。
- CTA 按钮组件命名体现动作，不用 `Button1`、`CardA` 这类无语义名字。

## 4. 页面与组件边界

页面文档必须说明：

- 哪些区块是页面定制内容
- 哪些区块应该抽成可复用组件
- 哪些交互状态属于页面层而不是单个组件层

建议做法：

- 视频是否可播放、页面是否可进入下一步，这类判断放页面层。
- 选项按钮高亮、禁用、hover 态，这类细节放组件层。

## 5. 文案消费规则

- 主标题、摘要、问题文案、CTA 文案都从配置读取。
- 前端不能根据 `branch_id` 自行生成展示语义。
- 如果字段缺失，只能走 fallback 文案，不允许拼接半残内容。

## 6. 抽象时机

当两个及以上页面出现以下任一情况时，应抽象公共组件：

- 区块结构一致，只换内容
- 状态层一致，只换提示文案
- 行为模式一致，只换跳转目标

如果只有视觉近似，但状态逻辑不同，不强行抽象。
