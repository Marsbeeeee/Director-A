# State Handling

所有 Demo Page 都必须显式处理四类状态：`loading`、`fallback`、`error`、`success`。不允许默认白屏，不允许只处理成功态。

## 1. 状态优先级

统一优先级：

`error > loading > fallback > success`

说明：

- `error` 表示请求失败、配置非法、资源不可用。
- `loading` 表示资源或配置仍在准备。
- `fallback` 表示页面可展示，但依赖内容不完整。
- `success` 表示主流程和主素材都可正常使用。

## 2. Loading

要求：

- 首屏 300ms 内给出可见反馈。
- 如果主素材是视频，loading 时先展示封面骨架和页面基础文案。
- 超过 3 秒仍未完成，补充“正在准备内容”说明。

禁止项：

- 只显示纯 spinner，没有上下文。
- 因等待视频而隐藏标题、摘要和 CTA。

## 3. Fallback

适用场景：

- 无视频，只有封面图
- 无真实品牌数据，只有占位内容
- 无完整台词，只有剧情摘要

要求：

- 页面主体仍然可理解。
- 主 CTA 仍然可用，除非后续流程确实依赖缺失数据。
- fallback 文案必须说明当前展示的是“封面版”“摘要版”或“占位版”。

## 4. Error

适用场景：

- 配置缺字段
- 接口失败
- 资源路径失效

要求：

- 页面展示可读错误说明，不暴露底层报错堆栈。
- 至少提供一个恢复动作：`重试`、`返回上一页`、`进入静态演示版` 三选一。
- 失败时记录错误类型，便于后续排查。

推荐错误提示格式：

- 标题：`页面暂时无法加载`
- 说明：`剧情素材未准备完成，当前无法进入视频版演示。`
- 动作：`重试加载`

## 5. Success

要求：

- 页面主视觉、剧情信息和下一步行为同时可见。
- CTA 不得被次要说明淹没。
- 如果存在自动跳转，页面上必须能看出“何时跳”“跳到哪”。

## 6. 事件埋点建议

每个页面至少记录：

- `page_view`
- `page_load_success`
- `page_load_error`
- `fallback_rendered`
- `primary_cta_clicked`

命名可以按项目调整，但同类页面必须一致。
