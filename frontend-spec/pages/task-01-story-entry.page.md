# Task 01 Page Spec: 恋综主剧情入口页

## 1. 基本信息

- 页面名称：恋综主剧情入口页
- 所属任务（Task ID）：Task 01
- 状态：`Planned`
- 负责人：Product / Frontend
- 关联任务文档：`tasks/task_01_story_entry.md`
- 最后更新：2026-04-24

## 2. 页面目标（Goal）

### 解决的问题

- 让用户在进入 Demo 的第一屏就明确知道：自己进入的是一段恋综剧情，不是品牌广告落地页。
- 建立后续互动节点的剧情前提，让“送香水 / 送口红”的选择有上下文，而不是孤立按钮。
- 给评审一个可直接开讲的入口页面，减少演示时额外解释成本。

### 第一感知

- 用户进入页面后的第一反应应是：`这是一个恋综约会前夕的剧情现场，马上要替男嘉宾做决定。`
- 页面必须在 3 秒内传达的信息是：`男嘉宾正在准备第一次正式见面，核心问题是该带什么礼物。`

## 3. 用户流程（User Flow）

1. 进入页面：
   - 用户打开 Demo 后直接进入该页面。
   - 页面先显示节目标题、场景封面或视频首帧，以及一句剧情摘要。
2. 观看或理解页面内容：
   - 用户看到男嘉宾正在准备第一次正式约会。
   - 页面同步展示问题文案：`第一次正式见面，他该带什么礼物？`
3. 触发主行为：
   - 用户点击 `开始观看`，或等待开场片段自动播放结束。
   - 系统准备进入下一步互动节点。
4. 获得反馈：
   - 点击后按钮进入 loading 态，文案变为 `正在进入选择节点`。
   - 如果是自动推进，页面显示短暂过渡提示：`剧情即将进入观众决策时刻`。
5. 跳转或继续后续流程：
   - 页面跳转到 `Task 02` 对应的礼物选择互动节点页。
   - 同时记录一次 `demo_entry` 事件，用于后续行为分析。

## 4. 页面结构（UI Structure）

### 区块 1

- 名称：背景区
- 作用：承载恋综氛围，建立“节目现场感”
- 必须展示的内容：视频首帧或封面图、环境氛围、人物主体

### 区块 2

- 名称：标题信息区
- 作用：快速说明当前节目和剧情节点
- 必须展示的内容：节目标题、场景标题、当前阶段标签，例如 `第一次正式约会前`

### 区块 3

- 名称：剧情摘要区
- 作用：用最短文本补足上下文，避免用户只看画面猜剧情
- 必须展示的内容：男嘉宾处境、约会背景、当前任务摘要

### 区块 4

- 名称：问题区
- 作用：明确抛出即将发生的互动问题
- 必须展示的内容：`第一次正式见面，他该带什么礼物？`

### 区块 5

- 名称：CTA 区
- 作用：承接进入下一步的主行为
- 必须展示的内容：主按钮 `开始观看`，辅助说明 `将进入观众选择节点`

## 5. 交互逻辑（Interaction）

| 用户操作 | 触发条件 | 系统反馈 | 后续结果 |
| --- | --- | --- | --- |
| 点击 `开始观看` | 页面成功渲染后 | 按钮进入 loading，显示 `正在进入选择节点` | 跳转到 Task 02 页面 |
| 等待开场片段播放结束 | 页面配置 `auto_advance=true` | 显示 1 条过渡提示 | 自动跳转到 Task 02 页面 |
| 点击 `重试加载` | 当前处于 error 状态 | 重新请求页面配置或重新挂载媒体资源 | 成功后回到 success，失败则停留 error |
| 点击 `进入静态版` | 视频不可用但有封面图 | 切换为封面 + 摘要展示 | 保持当前页可用，并允许继续进入 Task 02 |

要求：

- 页面不允许出现“用户无操作但流程卡住”的状态。
- 自动推进和按钮推进只能二选一作为默认主路径；本页建议保留按钮推进，自动推进作为可选配置。

## 6. 状态设计（State Handling）

### loading 状态

- 触发条件：页面首次加载、视频资源准备中、配置请求尚未完成
- 页面表现：先显示页面背景骨架、标题占位、摘要占位和不可点击的主按钮
- 是否允许操作：不允许主 CTA 触发跳转；允许展示只读剧情文案占位

### fallback 状态

- 触发条件：无视频、只有封面图；无完整旁白、只有摘要文本
- 页面表现：显示静态封面、节目标题、剧情摘要、问题文案和 `继续进入选择` 按钮
- 是否允许继续主流程：允许，且不降低主流程优先级

### error 状态

- 触发条件：必填字段缺失、封面与视频资源都不可用、页面配置读取失败
- 页面表现：显示错误标题、原因说明和两个动作按钮 `重试加载` / `进入静态版`
- 恢复动作：优先重试；若仍失败但存在最低限度文本配置，则允许进入静态版

### 成功状态

- 达成条件：标题、摘要、问题文案、主 CTA 均可展示，且至少有视频或封面图一种素材可用
- 页面表现：完整呈现恋综入口画面，并给出清晰的下一步入口
- 成功后的默认下一步：进入 `Task 02` 礼物选择互动节点

## 8. 技术实现建议（Tech Stack）

- 页面框架：React
- 样式方案：Tailwind 或 CSS Modules，优先使用语义化 token，不直接写散落颜色值
- 媒体能力：需要视频容器，至少支持 `poster + play / fallback`
- 动效能力：需要页面淡入和 CTA 点击反馈；不需要复杂时间轴动画
- 是否可使用 AI 生成稿：可以使用 `v0` 或 `Stitch` 生成低保真初稿，但最终结构和文案以本页文档为准
- 开发约束：
  - 问题文案必须由配置提供，不能硬编码在组件里
  - 页面必须同时兼容桌面横屏和移动端竖屏
  - 视频加载失败时必须自动切到封面版，不允许白屏

## 9. 可扩展性（Scalability）

- 多场景支持方式：把 `title`、`scene_summary`、`question_text`、`video_url` 抽为场景配置，不把恋综内容写死在组件内
- 多节目支持方式：增加 `show_id`、`show_name`、`scene_tags` 字段，通过配置切换不同节目入口页
- 需要抽象的组件：`DemoPageShell`、`SceneHero`、`StorySummary`、`PageFooterAction`
- 后续可能新增的字段：`countdown_seconds`、`voiceover_text`、`character_cards`、`background_music`
- 当前不抽象的部分：恋综专属文案语气和节目标签展示，可以先保留在 Task 01 页面层

## 10. 开发落地备注

- 推荐拆分组件：`StoryEntryPage`、`SceneHero`、`StorySummary`、`EntryCTA`、`StatusLayer`
- 推荐 mock 文件：`mock/story-entry.page.json`
- 验收重点：
  - 第一屏是否像剧情入口而不是广告封面
  - 无视频时是否仍能清楚表达剧情问题
  - 点击或自动推进后是否稳定进入 Task 02
